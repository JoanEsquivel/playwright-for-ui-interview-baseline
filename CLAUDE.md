# Playwright Project — Architecture Rules

These rules apply to every task in this project. They are always active — not just when a skill is invoked.

---

## Project at a Glance

- **Base URL:** `https://www.saucedemo.com/` (set in `playwright.config.js`)
- **Credentials:** `process.env.email` / `process.env.password` (from `.env` via dotenv)
- **Package manager:** `pnpm`
- **Run tests:** `pnpm exec playwright test`
- **Auth storage:** `.auth/auth.json` (generated by `tests/auth.setup.js`, gitignored)

## Core Rules — Non-Negotiable

### Rule 1: Assertions belong in tests, not in page objects or E2E utils

Page objects and the E2E utility class are **action helpers**. They navigate and interact. They do **not** own test outcomes.

```javascript
// WRONG — assertion inside page object action method
async submitLoginInForm(username, password) {
    await this.usernameInput.fill(username);
    await this.passwordInput.fill(password);
    await this.loginInButton.click();
    await expect(this.page).toHaveURL('/home'); // belongs in the test
}

// CORRECT — action only in page object
async submitLoginInForm(username, password) {
    await this.usernameInput.fill(username);
    await this.passwordInput.fill(password);
    await this.loginInButton.click();
}

// CORRECT — assertion lives in the test
test('should redirect to home after login', async ({ loginPage, homePage }) => {
    await loginPage.submitLoginInForm(process.env.email, process.env.password);
    await expect(homePage.heading).toBeVisible();
});
```

### Rule 2: Three allowed exceptions for assertion-like calls inside page objects / E2E utils

These are **guards**, not test assertions. They ensure the page is ready before proceeding.

| Exception | Purpose | Pattern |
|-----------|---------|---------|
| Dynamic wait before interaction | Confirm element is interactable | `await element.waitFor({ state: 'visible' })` |
| `waitLoad()` navigation guard | Confirm page transition happened | `await this.someElement.waitFor({ state: 'visible' })` inside `test.step()` |
| Internal `expect()` guard in E2E utils | Confirm a multi-step flow advanced to the next page | `await expect(element).toBeVisible()` used only as a bridge, not a test outcome |

Always wrap these in `test.step()` inside `waitLoad()` — keeps them visible in reports without polluting test logic.

### Rule 3: Always import test/expect from the index fixture in spec files

```javascript
// CORRECT in tests/*.spec.js
import { test, expect } from '../fixtures/index.fixtures'

// CORRECT in tests/e2e/*.spec.js
import { test, expect } from '../../fixtures/index.fixtures'

// WRONG — never import directly from @playwright/test in spec files
import { test, expect } from '@playwright/test'
```

Exception: page object files import `test` from `@playwright/test` **only** to access `test.step()` inside `waitLoad()`.

### Rule 4: Locators use semantic selectors with `.describe()`

```javascript
// Preferred
this.loginButton = page.getByRole('button', { name: 'Login' }).describe('Login button');
this.usernameInput = page.getByRole('textbox', { name: 'Username' }).describe('Username input');

// Acceptable when no semantic alternative exists
this.spinner = page.locator('[data-testid="loading-spinner"]').describe('Loading spinner');
```

### Rule 5: Never hard-code credentials

```javascript
// WRONG
await loginPage.submitLoginInForm('standard_user', 'secret_sauce');

// CORRECT
await loginPage.submitLoginInForm(process.env.email, process.env.password);
```

## File-by-File Responsibilities

| File | What it owns | What it must NOT contain |
|------|-------------|--------------------------|
| `pages/*.js` | Locators (constructor), `load()`, `waitLoad()`, action methods | `expect()` assertions, test data |
| `utils/e2e.js` | Multi-step workflow methods | `expect()` assertions (except navigation guards via `waitLoad()`) |
| `fixtures/page.fixtures.js` | One fixture entry per page object | Business logic, actions |
| `fixtures/e2e.fixtures.js` | One `e2e` fixture entry | Multiple fixture types |
| `fixtures/index.fixtures.js` | `mergeTests()` result, re-exports `expect` and `request` | Any logic |
| `tests/*.spec.js` | `describe`, `beforeEach`, `test` blocks, all assertions | Direct `page.goto()` calls — use page objects |
| `tests/e2e/*.spec.js` | Same as above, E2E-scoped | Direct `page.goto()` calls — use page objects |
| `tests/auth.setup.js` | Auth flow via `e2e.login()`, saves `storageState` | Test assertions beyond navigation guards |
| `data/test.json` | Static test data as JSON | Credentials (those go in `.env`) |
| `playwright.config.js` | `baseURL`, browser projects, `storageState`, env vars | Test logic |

## Common Mistakes to Avoid

**Using `page.goto()` directly in tests**
```javascript
// WRONG
test('...', async ({ page }) => { await page.goto('/checkout'); });

// CORRECT
test('...', async ({ checkoutPage }) => { await checkoutPage.load(); });
```

**Importing from `@playwright/test` in spec files**
```javascript
// WRONG
import { test, expect } from '@playwright/test';

// CORRECT
import { test, expect } from '../fixtures/index.fixtures';
```

**Hard-coding credentials**
```javascript
// WRONG
await loginPage.submitLoginInForm('standard_user', 'secret_sauce');

// CORRECT
await loginPage.submitLoginInForm(process.env.email, process.env.password);
```

**Assertions in action methods**
```javascript
// WRONG
async addItemToCart() {
    await this.addButton.click();
    await expect(this.cartBadge).toHaveText('1'); // belongs in test
}

// CORRECT
async addItemToCart() {
    await this.addButton.click();
}
```

**Skipping `waitLoad()` after `load()`**
```javascript
// WRONG — flaky, page may not be ready
test.beforeEach(async ({ loginPage }) => {
    await loginPage.load();
});

// CORRECT
test.beforeEach(async ({ loginPage }) => {
    await test.step('Load login page', async () => {
        await loginPage.load();
        await loginPage.waitLoad();
    });
});
```

**Creating a page class without wiring it into fixtures**
A page class in `pages/` that has no entry in `fixtures/page.fixtures.js` is never available as a fixture parameter. Always complete both steps.
